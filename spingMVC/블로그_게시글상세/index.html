1. 테이블 설계

넘나 미개한 테이블 설계… 아직 컬럼 타입에 대한 길이 설정을 어떻게 해야할지 잘 모르겠다.
차차 공부하여 적절하게 수정하기로 하고 아래와 같이 테이블을 만들고 테스트 데이터를 몇개 집어넣었다.

CREATE TABLE `POST` (
  `POST_ID` int(11) NOT NULL AUTO_INCREMENT COMMENT '게시글 ID',
  `POST_TITLE` varchar(255) NOT NULL COMMENT '게시글 제목',
  `POST_CONTENT` text DEFAULT NULL COMMENT '게시글 본문 내용',
  `POST_USERNAME` varchar(20) NOT NULL COMMENT '게시글 작성자',
  `POST_DATETIME` datetime DEFAULT NULL COMMENT '게시글 작성 일자',
  `POST_UPDATETIME` datetime DEFAULT NULL COMMENT '최종 수정 시간',
  `POST_SECRET` tinyint(4) NOT NULL DEFAULT 0 COMMENT '비밀글 여부',
  `POST_HIT` int(11) NOT NULL DEFAULT 0 COMMENT '조회수',
  PRIMARY KEY (`POST_ID`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8 COMMENT='게시글';

2. PostVO

Lombok을 설치하여 VO를 간소화, 처음에 테이블 컬럼명과 네이밍이 틀려서 어떻게 해야하나 고민했는데 조금 검색하다보니 Mybatis 설정을 통해 해결 하였는데 너무나 신박하다. 까먹지 않기 위해 따로 정리를 하자!

@Data
public class PostVO {

	private int postId;
	private String postTitle;
	private String postContent;
	private String postUserName;
	private Date postDatetime;
	private Date postUpdatetime;
	private boolean postSecret;
	private int postHit;
	
}

3. PostMapper

게시글 한건을 조회하기 위한 쿼리를 테스트 해보고 Mapper를 미리 만들었다.
selectPost는 한건을 조회하기 위한 쿼리이고 updatePostHit은 조회수 증가를 위한 쿼리이다.
게시글 비공개 여부에 대한 조건은 나중에 사용자를 추가하면서 처리할 생각..

<mapper namespace="com.jeong.blog.mapper.postMapper">

	<select id="selectPost" parameterType="int" resultType="com.jeong.blog.web.vo.PostVO">
		SELECT POST_ID, POST_TITLE, POST_CONTENT, POST_USERNAME, POST_DATETIME, POST_UPDATETIME, POST_SECRET, POST_HIT
		FROM jlog.POST
		WHERE POST_ID = #{postId};
	</select>

	<update id="updatePostHit" parameterType="com.jeong.blog.web.vo.PostVO">
		UPDATE jlog.POST
		SET POST_HIT=POST_HIT + 1
		WHERE POST_ID=#{postId};
	</update>
	
</mapper>

4. PostController

블로그 글을 보기위해 컨트롤러를 작성하고 @PathVariable 어노테이션을 이용해 postId값을 받아 서비스를 호출하고 결과로 넘어온 객체를 그대로 뷰에 전달하였다.

@Controller
public class PostController {
	
	private final Logger logger = LoggerFactory.getLogger(this.getClass());
	
	@Autowired
	PostService postService;
	
	@RequestMapping(value = "/post/view/{postId}", method = RequestMethod.GET)
	public String view( @PathVariable int postId, Model model ) {
		
		logger.debug("========== REQUEST START : /post/{postId} ==========");
		logger.debug(">>>>> postId : {}", postId);
		
		model.addAttribute("post", postService.getPost(postId));
		
		return "post/view";
	}
	
}

5. PostService & PostServiceImpl

아직은 특별히 처리해야할 로직이 없어 postId에 해당하는 게시글을 하나 가져와서 PostVO 객체를 리턴시키는 viewPost 메서드를 추가한다.

public interface PostService {
	
	public PostVO viewPost(int postId);
	
}

실제 viewPost 메서드를 구현한다.

@Service
public class PostServiceImpl implements PostService {
	
	private final Logger logger = LoggerFactory.getLogger(this.getClass());
	
	@Autowired
	PostDAO postDAO;
	
	@Override
	public PostVO viewPost(int postId) {
		
		logger.debug("========= SERVICE START : viewPost ==========");
		
		return postDAO.viewPost(postId);
	}

}

6. PostDAO

앞서 postMapper에 작성된 쿼리를 사용하여 조회수를 먼저 증가시키고 해당하는 게시글을 한건 가져온다.
여기서 하나 고민이 생긴게 현재 상태는 조회수가 무조건 증가되어버리는데 이미 조회된 게시글에 대한 조회수 증가를 어떻게 처리해야할지 고민이다..

@Repository
public class PostDAO extends AbstractDAO {
	
	private final Logger logger = LoggerFactory.getLogger(this.getClass());
	
	private static String namespace ="com.jeong.blog.mapper.postMapper";
	
	public PostVO viewPost(int postId) {
		
		logger.debug("========= DAO START : getPost ==========");
		
		// 조회수 증가
		update(namespace + ".updatePostHit", postId);
		
		return (PostVO) selectOne(namespace + ".selectPost", postId);
	}
	
}

정리하기

전체적인 흐름을 한번 생각해보자 컨트롤러에 매핑된 주소로 게시글 번호를 넘겨 컨트롤러에서 서비스를 호출고 DAO를 통해 DB에 접근하여 리턴받은 결과를 통해 다시 뷰에 보내서 페이지를 만든다.
생각보다 수월하게 진행이 되었고 재미있다 코딩과는 별개로 아직은 비즈니스 로직같은 각 계층에 대한 개념이 조금 명확하지 않은것같다 시간날때 틈틈히 많은 글들을 읽어보자.
